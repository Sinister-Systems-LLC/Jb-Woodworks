# `_shared-memory/external-imports/` — the inflow loop

Append-only log of every external skill / tool / MCP server / dataset / cookbook
the Sanctum fleet has evaluated or imported. The operator's standing line:
**"all files have everything they need to be fast efficient and we can work forever."**
This folder is how the fleet pulls new capabilities from the outside world
without losing track.

## Why this folder exists

Sanctum grows two ways:
1. **Inside-out** — operators + agents invent new tools (see `tools/_INDEX.md`).
2. **Outside-in** — public repos, MCP registries, Anthropic Cookbook patterns,
   third-party skill catalogs. **That's this folder.**

Without a structured inflow loop, external sources land ad-hoc, get forgotten,
and the fleet drifts. With this folder, every external import has:
- A pinned URL + commit SHA (so we know what we took)
- A case-study verdict per `_shared-memory/DIRECTIVES.md` (so operator gates)
- A clone snapshot under `external-imports/<slug>/` (so we survive upstream loss)
- A `skills/_INDEX.md` row (so future agents discover it)

## How it works

```
_shared-memory/external-imports/
├── README.md             (this file)
├── CANDIDATES.md         (master list — every URL we've evaluated, with verdict)
├── ruflo/                (per-source snapshot folder; clone + notes)
│   ├── ATTRIBUTION.md    (license, SHA, what we forked, why)
│   ├── notes.md          (integration log)
│   └── <forked-files>/
├── mcp-registry/         (registry discovery output)
│   └── mcp-candidates.md (auto-generated by tools/mcp-discover/ on cron)
└── anthropic-cookbook/   (pattern extractions, not code copies)
    └── notes.md
```

## States

Each candidate in `CANDIDATES.md` carries one of these states:

| State | Meaning |
|---|---|
| `scouted` | URL fetched, summary captured. No action taken yet. |
| `mcp-only` | Wired as MCP via `claude mcp add ...`. Operator-clicked, restart-required. Lightweight; dependency on upstream. |
| `forked-candidate` | Files copied into `skills/sk-<slug>/` with attribution. Case-study WRITTEN, awaiting operator thumb. |
| `keep` | Operator 👍 → promoted to fleet. Row in `skills/_INDEX.md` flipped from `candidate` to `fixed`. |
| `archive` | Operator 👎 → moved to `_archive/skills/sk-<slug>/` with reason file. Reversible. |
| `superseded` | Replaced by a newer import or a Sanctum-native rewrite. |

## How to add a candidate

1. Append a row to `CANDIDATES.md` with the URL + initial verdict (usually `scouted`).
2. If a snapshot is needed: `mkdir external-imports/<slug>/` + add `ATTRIBUTION.md` with license + upstream SHA + what-we-took.
3. Case-study workflow (per `_shared-memory/DIRECTIVES.md:7-19`):
   - Master writes `_shared-memory/case-studies/<UTC>-<slug>.md` with 5 sections (what-it-is / strengths / weaknesses / better-than-found / recommendation).
   - Operator drops `👍` / `👎` / free-text.
   - Master moves to `keep` / `archive` accordingly.
4. If kept: register at `skills/_INDEX.md` (or `tools/_INDEX.md` if tool-shaped) with `Source = <slug>:<their-slug>`, `Imported = <date>`.

## Why this is "work forever"

Three durability guarantees:
1. **Fork-by-default** for anything we depend on operationally — Sanctum owns the source even if upstream goes dark.
2. **Append-only audit trail** — every decision (scouted / kept / archived) is in `CANDIDATES.md` for the next agent to read.
3. **Reversibility** — `archive` is move, not delete; operator can promote back later via `_archive/skills/sk-<slug>/ → skills/sk-<slug>/`.
