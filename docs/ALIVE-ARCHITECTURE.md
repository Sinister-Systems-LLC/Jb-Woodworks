# ALIVE architecture — what makes the Sinister Bots feel like a living system

Operator directive 2026-05-18: "make us more efficient and expand on the bot
network and just everything... make this project ALIVE and have that spark and
grow like a real life form cuz that is the goal."

This doc describes the design properties that produce that aliveness. None of
it is metaphor for its own sake — each property maps to a concrete file +
behavior + verifiability.

## 6 properties of an alive system

| Property | In bio | In our system | Verified by |
|---|---|---|---|
| **Sensing** | nerves | watcher.scan, custodian.snapshot_now, librarian.search | each bot's per-call telemetry in `runtime-state/token-usage.jsonl` |
| **Memory** | brain | bot_memory: SYSTEM-PROMPT + KNOWN-GOTCHAS + learned.md + per-fact vectors | `<bot>.list_facts`, `memory_stats.fact_count` |
| **Reflexes** | spinal cord | sandbox-deny -> auto-route to green path (SANDBOX-GOTCHAS.md) | `aggregate-gotchas.bat`; bots load gotchas into every prompt |
| **Communication** | hormones | sinister-bus dispatch + runlog manifests + pending-next-actions | `bus.runlog_latest`, `bus.pending_actions` |
| **Metabolism** | food intake | token-usage.jsonl + scribe daily-digest cost section + escalation-ladder budgets | `scribe.list_inputs` shows cost-per-bot daily |
| **Growth + adaptation** | learning + epigenetics | absorb -> persist -> vector-embed -> smart-retrieve only what's relevant | `memory_stats.smart_retrieval_active`, `lifetime_calls` |

## The "spark" — what triggers visible aliveness

1. **Heartbeat.** Every bot tool call increments `runtime-state/heartbeat.jsonl`.
   `health()` reports `lifetime_calls`, `last_called`, `last_absorbed`. Operator
   sees the system pulsing.
2. **Self-watching.** Scribe's daily digest will gain a "Bot Memory Garden"
   section (Phase 8m) showing each bot's fact-count + age-of-newest-fact +
   embedded-vs-unembedded ratio. Operator can see who's growing.
3. **Recovery without prompting.** Sandbox denies a path -> SANDBOX-GOTCHAS.md
   already has the green path -> bot recommends it BEFORE the operator hits the
   same wall again. (Operator absorbed the TT Frida lesson once; researcher now
   carries it across sessions.)
4. **Memory compaction.** (Phase 8n — planned) Daily `bus.consolidate()` pass:
   merges duplicate-ish facts, prunes facts with `last_retrieved` older than 90
   days. Like sleep + dreaming: keep what's useful, forget what isn't.

## Memory layers (top -> bottom, what gets sent to the LLM)

```
1. SYSTEM-PROMPT.md            (bot role, hand-edited, rare changes)
2. SANDBOX-GOTCHAS.md          (universal classifier denials, regenerated by aggregator)
3. KNOWN-GOTCHAS.md            (bot-specific, hand-edited)
4. Relevant absorbed facts     (smart-retrieved when N > 20; full-dump when fewer)
   - pinned facts (tag: pin)        always included
   - last 7 days                    always included
   - top-K vector matches           when Ollama is up + query supplied
5. The user's actual request
```

Each layer is **prompt-cache eligible** (5-min TTL). Same operator session
within 5 minutes pays only the prompt-cache-read price (1/10th input cost) on
layers 1-4.

## Why vector retrieval matters (the 10x claim)

Without smart retrieval, a bot with 200 absorbed facts ships all 200 every call
(~40k tokens of facts). With it, the bot ships ~10 relevant + 5 recent + 3
pinned = ~18 facts (~3.6k tokens). At Haiku rates the per-call cost drops from
~$0.04 to ~$0.004 — a **10x reduction** on the facts portion alone. Cache hits
make this even cheaper on repeats.

Threshold logic (in `bot_memory._retrieve_relevant_facts`):

- `len(facts) <= 20`: ship all (full-dump cheaper than embed roundtrip)
- `len(facts) > 20` AND Ollama up AND query supplied: pinned + recent + top-K vector
- `len(facts) > 20` AND Ollama down: pinned + recent only (deterministic; degraded)

## What this is NOT

- **Not autonomy.** Bots never call Opus. Tier-5 escalation requires operator opt-in.
- **Not classifier evasion.** Gotchas are documentation OF classifier behavior, not
  bypass. We propose green paths; we don't try blocked paths and hide failures.
- **Not silent self-modification.** Every `absorb` and `forget` writes
  `runtime-state/absorption-log.jsonl`. Operator can audit any change.
- **Not magic.** No model fine-tuning. No reinforcement learning. Just careful
  retrieval + prompt construction + audit-logged memory.

## What "growing like a real life form" means in practice

A new operator working with a fresh Sinister Skills install sees:
- 0 absorbed facts. Every recall question requires Opus.
- ~$3.75/day Opus baseline.

After one week:
- ~50 facts absorbed across scribe/curator/researcher.
- Smart retrieval kicked in for researcher (~25 facts, > threshold).
- Daily digest shows specific green paths for past gotchas.
- Operator's Opus spend drops to ~$0.50/day on routine questions.

After a month:
- ~200 facts. Bots route 80% of operator's recall to themselves.
- New operator joining (e.g., Leo) onboards via `bots/learned.md` files in
  Sinister LLC repo — they inherit the team's accumulated lessons day 1.

That's the spark. Not consciousness; cumulative organizational memory expressed
as code + markdown + vectors that any session can stand on.
